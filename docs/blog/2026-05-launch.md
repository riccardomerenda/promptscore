# PromptScore: ESLint, but for prompts. And now in your CI.

> Draft. Save this file or cross-post to dev.to / Hashnode / your own blog. Reading time ~6 minutes.

If you write JavaScript, ESLint catches an entire class of mistakes before they ship: the `===` you forgot, the unused import, the dead branch, the `any` that should have been a real type. It does not run your code; it reads the source and tells you what is structurally wrong.

If you write **prompts** for an LLM, you have nothing comparable. You write a prompt, you send it, the model responds, and either the answer is good or it is not. If it is not, the feedback loop is opaque: maybe the prompt is ambiguous, maybe the format spec is missing, maybe the role is undefined, maybe you stacked three contradictory instructions, maybe the model is just having a bad day. You iterate, you tweak, you ship a version that mostly works, and the prompt sits in production with no one reviewing it the way they would review a function.

That is the gap [PromptScore](https://promptscore.dev) tries to close. It is a static analyzer for prompts. It does not call a model. It reads the prompt as text, scores it 0&ndash;100, and tells you what is structurally missing or weak, with concrete fixes you can paste back into the prompt.

## What it actually checks

The current release ships **twelve deterministic rules** plus an optional, opt-in LLM-backed review. The deterministic ones are fast, offline, and private &mdash; nothing leaves the machine.

A subset to give you the flavor:

- `missing-task` &mdash; is there an unambiguous instruction telling the model what to do? This is the highest-weight rule and the only one that fires as a hard error by default. If a model can&rsquo;t identify the task, the rest of the prompt is wasted.
- `no-output-format` &mdash; does the prompt say what shape the answer should take? JSON schema, bullet list, single sentence, markdown table?
- `no-examples` &mdash; few-shot examples dramatically improve consistency on classification, extraction, and formatting tasks.
- `no-role` &mdash; assigning a role focuses the model and sets expectations for tone and expertise.
- `ambiguous-negation` &mdash; negative instructions ("don't do X") are less effective than positive ones ("do Y instead"). Heavy stacking of negations correlates with regressions.
- `vague-instruction` &mdash; words like "good", "proper", "appropriate" don&rsquo;t give the model a measurable target.

Each rule, when it fails, produces three things: a **message** explaining the issue, a **fix suggestion** in plain English, and &mdash; for seven of the deterministic rules and the five non-generic LLM issue types &mdash; a **rewrite snippet** you can literally paste into the prompt.

For example, if `no-output-format` fires, the rewrite snippet looks like:

```text
<output_format>
  Return a <JSON object | bullet list | markdown table | single paragraph>.
  Required fields / sections: <list>.
  Do not include: <what to omit>.
</output_format>
```

You don&rsquo;t even have to think about what an output-format section should look like. You paste, fill the placeholders, done.

## A real example

Take a prompt that looks fine on a quick read:

```text
You are a senior writer.
Make this article better, but keep it concise.
Return it in markdown.
Do not be verbose.
Do not return JSON.
```

Run it through the analyzer (in the browser at [promptscore.dev](https://promptscore.dev), or via the CLI):

```bash
npx @promptscore/cli analyze prompt.txt
```

You get something like:

- **Overall: 47/100**
- `missing-task` &mdash; "Make this article better" is not a task, it is a wish. The rule offers a rewrite: `Your task is to <verb> <object>. Specifically: <what success looks like>.`
- `vague-instruction` &mdash; "better" and "concise" are quality terms without measurable targets. The fix suggests replacing with concrete criteria like "&le; 200 words" or "explicit citations".
- `ambiguous-negation` &mdash; three "do not" lines stacked. Models follow positive instructions more reliably; the fix recommends reframing as "do Y" patterns.
- `no-examples` &mdash; for a rewrite task, one or two before/after examples would dramatically improve consistency.

Each of those is a finding you would have caught yourself eventually, after the prompt had been in production for a week and the model had returned three bad outputs. Catching them at write-time is roughly the same value proposition as type-checking.

## Now in your CI

The release that just shipped &mdash; `v0.4.8` &mdash; adds an **official GitHub Action**. Three lines into any workflow:

```yaml
- uses: actions/checkout@v6
- uses: riccardomerenda/promptscore@main
  with:
    inputs: prompts/
    model: claude
    format: markdown
    fail-on: warning
```

The Action runs the CLI against the directory you point it at, exits non-zero when findings cross the configured severity, and (when `format: markdown`) appends the rendered report to the GitHub Actions job summary so reviewers see findings inline in the GitHub UI without opening the workflow log.

If your team checks prompts into the repo &mdash; whether for an agent, a classification job, a content pipeline, or any other LLM-backed feature &mdash; you can now block PRs that introduce ambiguous, ungrounded, or scope-undefined prompts the same way you block PRs that introduce unused imports.

## What this is not

PromptScore is not an LLM evaluation framework. It does not measure output quality. It does not run your prompt against a model and grade the answer. That is a different problem, with a different answer (Promptfoo, Helicone, Phoenix, OpenAI Evals, &hellip;). PromptScore stays firmly on the *input* side, where deterministic structural analysis is fast, free, and reliable.

It is also not trying to be an agent platform. Tracing, output evaluation, and runtime monitoring are out of scope by design.

## What&rsquo;s next

The deterministic engine is done for v0.4. The next priorities are real-world pickup &mdash; the GitHub Action is the obvious distribution lever &mdash; followed by hosted features for teams (shared policies, report history, collaboration) on the v0.5 timeline.

If you write prompts that other people depend on, try it on one of them. Five minutes:

```bash
npx @promptscore/cli analyze your-prompt.txt --model claude
```

The browser analyzer at [promptscore.dev](https://promptscore.dev) is the same engine if you want a click-and-paste version first.

The repo lives at [github.com/riccardomerenda/promptscore](https://github.com/riccardomerenda/promptscore). MIT, Node, no telemetry, no API calls, runs entirely on your machine. Issues and PRs welcome.
